# PRD-15: i18n Strategy (English + Korean)

> Parent: [PRD-00 Overview](./00-overview.md)  
> Status: Draft

---

## Purpose

Kaibase supports English and Korean as a **localization foundation**, with **English as the primary language**. The i18n infrastructure is designed to support additional languages in the future, but V1 ships with EN (primary) + KO (secondary). This document defines the internationalization strategy across UI, content, AI, and system infrastructure.

> **Why EN + KO?** English-first for global market reach. Korean for domestic market (Korea) and initial B2B customers. The i18n framework is generic enough to add any language later — EN/KO is the minimum viable localization scope.

---

## Scope

| Layer | EN/KO Support |
|-------|--------------|
| **UI labels & navigation** | Fully translated (EN + KO) |
| **System messages & errors** | Fully translated |
| **AI-generated content** | Configurable per workspace (EN, KO, or both) |
| **Q&A answers** | Language matches the question language |
| **Page content** | User-editable in any language; AI generates per policy |
| **Entity/concept names** | `name` (primary) + `name_ko` (Korean variant) |
| **Email digests** | Formatted per user's `locale` preference |
| **Slack/Discord messages** | Workspace default language |
| **MCP responses** | Requested language or workspace default |
| **Activity log** | Event types in system language; detail payloads in original language |

## i18n Framework

### Frontend

- **Library:** `react-i18next` with `i18next`
- **Namespace-based:** each major feature area has its own namespace
- **Lazy loading:** translation bundles loaded per namespace on demand
- **Fallback:** English as fallback language

### Translation File Structure

```
apps/web/src/locales/
  en/
    common.json        # shared labels (nav, buttons, status)
    pages.json         # page editor UI
    qa.json            # Q&A interface
    reviews.json       # review queue
    graph.json         # graph view
    settings.json      # settings screens
    notifications.json # notification text
    errors.json        # error messages
  ko/
    common.json
    pages.json
    qa.json
    reviews.json
    graph.json
    settings.json
    notifications.json
    errors.json
```

### Key Patterns

```typescript
// Simple label
t('common.save')         // EN: "Save" / KO: "저장"

// With interpolation
t('pages.created_by', { name: 'AI' })  // EN: "Created by AI" / KO: "AI가 생성"

// Plurals
t('reviews.pending_count', { count: 5 })  // EN: "5 pending reviews" / KO: "검토 대기 5건"

// Date/time formatting
format(date, 'PPP', { locale: ko })  // uses date-fns locale
```

## Language Selection

### User Level
- Each user has a `locale` preference ('en' | 'ko')
- Set during registration, changeable in profile settings
- Controls: UI language, notification language, date/time formatting

### Workspace Level
- Each workspace has a `default_language` ('en' | 'ko')
- Controls: default AI output language, channel output language
- AI config `summary_language` can be 'en', 'ko', or 'both'

### Per-Request
- Q&A answers match the detected question language
- MCP responses can specify language in request

## AI Content Generation

### Bilingual Summary Mode (`summary_language: 'both'`)

When workspace policy is set to 'both', AI generates:

```
## Summary / 요약

**English:**
This document describes the authentication architecture for Project Alpha...

**한국어:**
이 문서는 Project Alpha의 인증 아키텍처를 설명합니다...
```

### Page Content

- Pages store content in a single Y.Doc (not duplicated per language)
- AI generates content in the workspace's configured language
- If 'both', content includes both language versions in structured sections
- Entity names stored as `name` (primary/English) + `name_ko` (Korean)

### Search

- Full-text search supports both English and Korean
- PostgreSQL `tsvector` with appropriate dictionaries:
  - English: `english` dictionary
  - Korean: requires Korean text search configuration (e.g., `textsearch_ko` extension or custom dictionary)
- Semantic search (pgvector) uses multilingual embedding model

## Typography & Layout

### Font Stack

```css
--font-sans: 'Inter', 'Pretendard', -apple-system, BlinkMacSystemFont, 
             'Segoe UI', 'Noto Sans KR', sans-serif;
--font-mono: 'JetBrains Mono', 'D2Coding', monospace;
```

### Layout Considerations

- Korean text is generally wider than English for the same semantic content
- UI should use flexible widths, not fixed character counts
- Line height slightly higher for Korean (1.7 vs 1.5 for English)
- Mixed EN/KO content should use consistent baseline alignment

### Date/Time Formatting

| Locale | Example |
|--------|---------|
| EN | April 11, 2026, 3:30 PM |
| KO | 2026년 4월 11일 오후 3:30 |

Use `date-fns` with locale-aware formatting.

### Number Formatting

| Locale | Example |
|--------|---------|
| EN | 1,234.56 |
| KO | 1,234.56 (same convention) |

## Backend i18n

### System Messages

Backend returns error codes and message keys, not hardcoded strings:

```json
{
  "error": {
    "code": "POLICY_BLOCKED",
    "message_key": "errors.policy_blocked",
    "detail": { "rule_name": "Block low-confidence" }
  }
}
```

Frontend resolves `message_key` to the appropriate language string.

### Activity Log

Activity events store data in their original language. The UI renders event type labels in the user's locale but preserves detail content as-is.

---

## Translation Workflow

1. **English is the primary and source language** for all UI strings and AI content
2. Korean translations maintained in parallel but are secondary — missing KO is acceptable in early phases
3. New features must include EN translation keys before merge; KO keys can follow
4. Missing KO translations fall back to EN with a visual indicator
5. Translation keys follow dot-notation namespace pattern
6. The i18n framework is designed for future expansion (e.g., Japanese, Chinese) but V1 scope is EN + KO only
